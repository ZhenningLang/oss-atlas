# m3u8

一个面向 HLS（HTTP Live Streaming）`.m3u8` 播放列表的 Python 解析器与序列化器——把来自 URL、文件或字符串的播放列表加载成一个类型化对象模型，查看/修改 segment 与变体，再 dump 回去（RFC 8216）。

![m3u8 — 健康度雷达](../../assets/health/m3u8.zh.svg)

## 何时使用

你在做一个处理 HLS 流的东西——下载器、打包器、广告插入服务、流校验器——你需要以编程方式*读懂*一个 `.m3u8` 播放列表，而不是拿正则去抠。你 `pip install m3u8`，调 `m3u8.load('http://server/playlist.m3u8')`，拿到一个对象，其中 `playlist.segments`、`playlist.target_duration`、`playlist.playlists`（变体流）、key、byterange、discontinuity 都是可读可改的类型化属性。然后 `playlist.dumps()` 把合法的 HLS 再序列化出来。它覆盖了真实的标签面——`EXT-X-STREAM-INF`、`EXT-X-MEDIA`、`EXT-X-KEY`、`EXT-X-MAP`、SCTE-35 cue 标签（`EXT-X-CUE-OUT`/`CUE-IN`）、I-frame 播放列表、byterange——所以 master 和 media 播放列表都能干净解析。

当你的活儿专门是*操作 HLS manifest* 时，它就是首选：改写 URI、拼接广告标记、过滤清晰度档，或校验某个打包器输出是否合规——把这种文本格式变成你能推理的数据结构。[推断]

## 何时不用

- **你要的是媒体，不是 manifest。** m3u8 解析的是*播放列表文本*；它不下载分片、不解密、不解复用、不播放视频。请配一个 HTTP 客户端 / FFmpeg 来处理真正的媒体。
- **你不在 Python 里。** 它是 Python 库；别的运行时有自己的 HLS 解析器（hls.js 等）。
- **你需要 MPEG-DASH 或别的 ABR 格式。** 它专门针对 HLS/`.m3u8`；DASH 的 `.mpd` 完全是另一个解析器。
- **你需要对每个最新/草案 HLS 标签的确定覆盖。** 它跟随 RFC 8216（外加部分草案/SCTE-35），但更新的 `rfc8216bis` 增项或冷门厂商标签可能落后；依赖某标签前请对照当前代码核实。[未验证]
- **你在热路径上需要零分配解析。** 它是纯 Python 的便利对象模型；要极致吞吐的 manifest 解析，你得 profile 或下沉到更底层。

## 技术栈

- **语言：** 纯 Python；包裹 `.m3u8` 文法的对象模型。
- **API 形态：** `load`/`loads`（从 URI/文件/字符串）与 `dump`/`dumps`（到文件/字符串），配类型化对象——`M3U8`、`Segment`、`Key`、`Playlist`/`Media`（变体），外加 target duration、序列号、byterange、discontinuity 和 SCTE-35 cue 标签等属性。
- **规范覆盖：** RFC 8216 标签集，外加部分草案/`rfc8216bis` 与 SCTE-35 标签（见 README 的支持标签列表）。

## 依赖

- **运行时：** Python 加一个很小的依赖集合（`load(url)` 用的 HTTP 客户端，历史上有 `iso8601` 做日期解析）；经 `pip install m3u8` 安装。
- **外部：** 远程加载时需访问播放列表 URL 的网络——但没什么要搭的；解析本身不需要服务、数据库或媒体工具。
- **可选：** 你选用的 HTTP 客户端 / 一个媒体工具（FFmpeg），若你接下来要去抓取或处理播放列表引用的分片。

## 运维难度

**低。** 它是个纯 Python 解析库——`pip install` 后 import，没什么要部署或运维的。仅有的运维微妙处是：随 HLS 标签演进保持版本更新，以及对畸形/真实世界的播放列表做防御性处理（厂商 manifest 会以各种创造性方式违反规范）。无运行时基础设施、无数据存储；它只干一件事且不挡道。
